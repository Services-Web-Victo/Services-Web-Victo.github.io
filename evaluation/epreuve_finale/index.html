    Épreuve finale

    Énoncé de l’épreuve

    Vous devez développer seul un service web selon les spécifications décrites plus bas et l’intégrer dans un site Web.

    Contexte de réalisation

    • Le développement se fera durant le temps de classe et hors classe.
    • Date limite de remise : Lundi le 16 mai à 23 :59
    • La remise se fera par Github. Vous pouvez me faire un repo pour le service web et un autre pour le site ou bien me jumeler les deux dans le même.
    • Pour la base de données, me faire un dump avec création de tables et inclure le fichier dans le projet du service web.

    Spécification de l’épreuve

    Le service Web

    Le sujet du service web est à votre choix, mais il doit répondre aux critères suivants.

    1. La base de données doit être composée d’un minimum de deux tables.
    2. La structure du projet doit respecter une séparation entre la logique et les données.
    3. Vous devez créer les routes pour couvrir toutes les opérations CRUD sur une des tables de la base de données. Donc au minimum vous aurez 4 routes dans votre service web.
    4. Les routes doivent retourner le bon code d’état selon leur type et la situation. (200, 201, 403, etc.). Fiez-vous aux notes de cours pour la convention qu'on utilise dans le cours.
    5. Toutes les routes créées doivent être documentées selon la norme openAPI qu’on a vue en classe.
    6. Toutes les routes doivent être couvertes par des tests fonctionnels. (Voir section Tests plus loin)
    7. Au moins une des routes doit être protégée par un système d’authentification de votre choix (basic, avec JWT, etc.)
    8. Le service web doit être hébergé en ligne sur l’hébergeur de votre choix.

    Le client Web

    1. Le client web doit utiliser React pour l’affichage et la gestion des opérations avec le service web.
    2. Il permettra de se connecter à votre service web et d’interagir avec les routes.
    3. L’utilisateur devra pouvoir utiliser votre site web pour pouvoir afficher, ajouter, modifier et supprimer des données. Comme pour le service web, vous n’êtes pas obligé de couvrir toutes les tables possibles de votre base de données.
    4. Le site web doit aussi utiliser une seconde API que vous n’avez pas développée. Ça peut être une API publique (météo, Random User, Unsplash, etc.) ou bien l’api d’un autre élève.
    5. L’aspect esthétique ne sera pas évalué, je veut que ça fonctionne avant tout.

    Suivi de version

    1. Vous devez faire un suivi de version sur Github.
    2. Un premier push à votre repo distant doit être fait avant de commencer à développer.
    3. Ajoutez-moi comme collaborateur à votre repo.
    4. Utilisateur : FrechetteMathieu
    5. Email : frechette.mathieu@carrefour.cegepvicto.ca
    6. Vous devez faire des commits et des push de façon régulière (idéalement après chaque fonctionnalité terminée que vous codez).

    Tests

    Pour chaque route, vous devez tester au minimum les deux points suivants :

    • Le code de statut retourné est bon
    • Les valeurs de retours sont ce qui est attendu

    Vous devez effectuer vos tests directement dans l’api avec PHPUnit.
