    Utilisation de journaux d'évènements (Log files)

    Pourquoi utiliser des fichiers log?

    • C'est une bonne façon de débugger les erreurs
    • Ça nous permet de "voir" ce qui se passe dans notre application, de suivre comment les usagers l'utilisent
    • Pour garder une trace des opérations effectués par le serveur

    Il y a plusieurs choix possible pour créer des fichiers log et les utiliser. Nous allons utiliser la librairie Monolog pour php.

    Installation

    Vous pouvez retrouver la documentation de Monolog à cette adresse : https://seldaek.github.io/monolog/. L'installation se fait via composer avec l'instruction suivante : 

    composer require monolog/monolog

    Intégration à notre projet SLIM

    Dans le projet de départ base-slim-skeleton nous avons déjà une classe nommé LoggerFactory qui s'occupe de la création des fichiers et de l'écriture dans ceux-ci. Elle est situé dans le répertoire src/Factory Pour son utilisation nous avons aussi besoin d'un générateur de UUID qu'on peut installer aussi via composer : 

    composer require symfony/polyfill-uuid

    Normalement monolog et le générateur de uuid devrait déjà avoir été installé dans votre projet quand vous avec fait la commande composer update après avoir cloner le projet. Pour vérifier le tout allez dans le fichier composer.json, vous devriez avoir les lignes suivantes : 

    {
    "require": {
        "monolog/monolog": "^2.2",
        "symfony/polyfill-uuid": "^1.22",
        ...
    },
    ...
}

    Une fois les deux dépendances installées :

    1. Créez un répertoire nommé logs à la racine de votre application
    2. Ajouter le nom de se répertoire dans le fichier .gitignore pour exclure de vos commit Git les fichiers de log que vous allez créer.
    vendor/
logs/
    1. Assurez-vous d'avoir les lignes suivantes dans le fichier config/settings.php
    // Logger settings
$settings['logger'] = [
    'name' => 'app',
    // C'est ici qu'on indique dans quel répertoire sera les fichiers
    // à modifier si vous avez utiliser un autre nom
    'path' => __DIR__ . '/../logs',
    'filename' => 'app.log',
    'level' => \Monolog\Logger::DEBUG,
    'file_permission' => 0775,
];
    1. Dans le fichier config/container.php, ajoutez les lignes suivantes si elles n'y sont pas déjà
    <?php

use App\Factory\LoggerFactory; 
// ...

return [

    // ...

    LoggerFactory::class => function (ContainerInterface $container) {
        return new LoggerFactory($container->get('settings')['logger']);
    },

    // ...

];

    Si vous voulez "logger" les erreurs générés par SLIM automatiquement, modifiez le fichier config/middleware.php de la façon suivante : 

    // Remplacer cette ligne
$app->add(ErrorMiddleware::class);

// Par ces lignes
$loggerFactory = $app->getContainer()->get(\App\Factory\LoggerFactory::class);
// Le nom du fichier où les erreurs seront inscrites
$logger = $loggerFactory->addFileHandler('error.log')->createLogger();
$app->addErrorMiddleware(true, true, true, $logger);

    Utilisation dans notre projet SLIM

    Pour écrire dans un fichier de log depuis une classe de notre projet, on doit récupérer un objet LoggerFactory passé en paramètre dans le constructeur de la classe. On peut ensuite utiliser cette objet pour écrire le message souhaité.

    Exemple avec la classe HomeAction

    <?php

namespace App\Action;

use App\Factory\LoggerFactory; // À ajouter
// ...

final class HomeAction
{

    /**
     * @var LoggerInterface
     */
    private $logger;

    public function __construct(LoggerFactory $loggerFactory)
    {
        $this->logger = $loggerFactory
            // Le nom de fichier de log utilisé
            ->addFileHandler('hello.log')
            // On peut passer du texte en parametre ici qui identifiera
            // la ligne de log, sinon un UUID sera utilisé
            ->createLogger('MessageFromMath');
    }

    public function __invoke(
        ServerRequestInterface $request, 
        ResponseInterface $response
    ): ResponseInterface {

        // ...
        // C'est ici qu'on inscrit notre message
        $this->logger->info('Hello world');

        // ...
    }
}

    Le résultat sera inscrit dans le fichier hello-[date-du-jour].log

    [2022-02-11T05:52:53.347534+01:00] 73c96cc8-4c6c-42d7-8a39-593ab0f964c6.INFO: Hello World  
[2022-02-11T05:52:56.912031+01:00] MessageFromMath.INFO: Hello World

    Niveaux de sévérité des messages

    Quand on inscrit un message d'événement, on lui attache un niveau de sévérité. Ça nous permet de catégoriser les messages et même dans omettre selon la configuration. Les différents niveaux supportés par Monolog sont les suivants, trier par ordre d'importance :

    • DEBUG - Message de débogage
    • INFO - Message d'information
    • NOTICE - Il est arrivé un événement important
    • WARNING - Message d'avertissement mais pas d'erreur
    • ERROR - Message d'erreur mais qui ne requiert pas d'action immédiate
    • CRITICAL - Message de condition critique
    • ALERT - Erreur qui nécessite une action immédiate
    • EMERGENCY - Alerte rouge

    Pour chacun des niveaux, il y a une fonction du même nom à utiliser

    // Message d'information
$this->logger->info("Ceci est un message d'information");
// Message d'erreur
$this->logger->error("Ceci est un message d'erreur");

    Pour modifier le niveau minimum de sévérité, c'est-à-dire à partir de quel niveau le message sera inscrit dans le fichier, on doit changer la valeur de la clé level du tableau settings dans le fichier config/settings.php

    // Logger settings
$settings['logger'] = [
    'name' => 'app',
    'path' => __DIR__ . '/../logs',
    'filename' => 'app.log',
    'level' => \Monolog\Logger::DEBUG, // Changer la valeur de level ici
    'file_permission' => 0775,
];

    Si on change cette valeur pour 'level' => \Monolog\Logger::ERROR, dans l'exemple plus haut seulement le message d'erreur sera inscrit, pas le message "info". L'utilité principale est qu'on peut contrôler les messages à retenir quand notre projet est déployer sur le Internet.
