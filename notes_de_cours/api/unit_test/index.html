    Tester notre API

    Pourquoi tester notre application?

    Dès qu’on ajoute du code à notre application, on ouvre la porte à ajouter aussi de possible erreurs ou comportement qu’on avait pas prévu. Pour s’assurer d’avoir une application solide et plus facilement modifiable, c’est une bonne pratique de programmer des tests qu’on pourra lancer au besoin par un script.

    Plusieurs possibilités s’offrent à nous comme plate-forme de tests si on ne veut pas tout programmer nous-même. Dans le cadre du cours on va en voir 2 :

    • Postman, qui offre la possibilité d’inclure des tests scriptés en javascript.
    • PHPUnit, un framework php

    Tester avec Postman

    Dans l'interface de postman, on peut scripter des tests dans l'onglet Tests d'une requête.

    Les scripts de tests sont codés en javascript et seront lancés une fois la réponse de l’appel à l’api reçu. Chaque test aura la structure suivante : 

    pm.test("Description du test", function() {
    // Le script du test
})

    Le résultat des tests se retrouvent dans la section Response sous l’onglet Test Results.

    Exemple d'une série de test

    /* Structure de la réponse
[
    {
        "id": "54",
        "genreId": "1",
        "titre": "Fondation",
        "isbn": "9782070463619"
    },
]
*/

// La constante jsonData va contenir un objet json de la réponse
const jsonData = pm.response.json();

pm.test("La réponse retourne le code 200", function () {
    pm.response.to.have.status(200);
});

pm.test("La réponse est un tableau", function () {
    pm.expect(jsonData).to.be.an("array");
})

pm.test("Les items du tableau de la réponse ont les bonnes clés", function() {
    if(jsonData.length > 0){
        // Valide que le premier item du tableau books contient les clés id, genreId, titre et isbn
        pm.expect(jsonData[0]).to.have.all.keys('id', 'genreId', 'titre','isbn');
    }
})

pm.test("Les items du tableau de la réponse ont les bons types", function() {
    if(jsonData.length > 0){
        pm.expect(jsonData[0].id).to.be.a("string");
        pm.expect(jsonData[0].genreId).to.be.a("string");
        pm.expect(jsonData[0].titre).to.be.a("string");
        pm.expect(jsonData[0].isbn).to.be.a("string");
    }
})

    Notre premier test

    À l'aide de la route https://api.chucknorris.io/jokes/random testez que les clés url ou value soient bien présente dans la réponse.

    Pour plus d'information et d'exemples, consultez la documentation de Postman :

    • https://learning.postman.com/docs/writing-scripts/script-references/test-examples/

    Tester avec PHPUnit

    Installation et intégration dans un projet

    Faites l’installation de PHPUnit comme dépendance de développement avec la commande suivante

    composer require phpunit/phpunit --dev

    Chaque tests sera une classe PHP qui sera présente dans le répertoire tests/TestCase/ de notre application

    • Créer le répertoire ./tests/TestCase/ à la racine de l’application
    • Avec git bash vous pouvez faire la commande mkdir –p tests/TestCase

    Ouvrez de fichier composer.json et ajoutez les lignes suivantes à la suite du fichier 

    "scripts": {
    "test": "phpunit --configuration phpunit.xml --testdox",
    "test:coverage": "phpunit --configuration phpunit.xml --coverage-clover build/logs/clover.xml --coverage-html build/coverage"
}

    Configuration de PHPUnit

    Créez le fichier phpunit.xml à la racine de l’application. La configuration de PHPUnit se fera dans ce fichier. Copiez le code suivant dans le fichier.

    <?xml version="1.0" encoding="UTF-8"?>
<phpunit 
    xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" 
    bootstrap="vendor/autoload.php" 
    colors="true" 
    backupGlobals="false" 
    backupStaticAttributes="false" 
    xsi:noNamespaceSchemaLocation="https://schema.phpunit.de/9.3/phpunit.xsd">
  <coverage processUncoveredFiles="false">
    <include>
      <directory suffix=".php">src</directory>
    </include>
    <exclude>
      <directory>vendor</directory>
      <directory>build</directory>
    </exclude>
  </coverage>
  <testsuites>
    <testsuite name="Tests">
      <directory suffix="Test.php">tests</directory>
    </testsuite>
  </testsuites>
</phpunit>

    Modifiez le fichier .gitignore pour ajouter la ligne suivante build/.

    Exécuter les tests

    Pour lancer les tests, faire la commande suivante dans un terimal à la racine de l’application 

    composer test

    Pour lancer les tests avec une couverture du code, faire la commande suivante 

    composer test:coverage

    Après avoir lancé composer test:coverage, vous aurez accès à deux pages html de résumé des tests dans le répertoire build/coverage.

    Erreur avec la commande test:coverage

    Si vous utilisez AMPPS et que vous rencontrez l’erreur suivante Error: No code coverage driver is available , il vous faut éditer le fichier php.ini de la façon suivante : 

    // Modifiez ces lignes
;Xdebug
;zend_extension="{$path}\php-7.3\lib\php_xdebug.dll"
// Comme ceci
[Xdebug]
zend_extension="{$path}\php-7.3\lib\php_xdebug.dll"
// Et ajoutez les deux lignes suivantes à la suite
xdebug.remote_autostart = 1
xdebug.remote_enable = 1

    Pour accéder rapidement au fichier php.ini, affichez le tableau de bord de Ammps et à la ligne PHP cliquez sur Settings et ensuite Configuration.

    Ajouter un "trait" à notre projet

    Un trait est une sorte de classe abstraite qui nous permet de regrouper des fonctionnalités pour ensuite les utiliser dans des classes. On va ajouter un trait à notre projet pour faciliter l'execution des tests.

    • Dans votre projet, créez le répertoire tests\Traits
    • Téléchargez le fichier AppTestTrait.php et copiez-le dans le répertoire nouvellement créé.

    Comme vous pouvez voir, le fichier contient déjà sont lots de fonctions qu'on va pouvoir utiliser pour l'éxécution de nos tests. Vous n'avez pas besoin de "comprendre" tous les fonctions, le but ici étant de se concentrer sur l'écriture des tests.

    Notre premier test unitaire

    Un test unitaire est un test effectué sur une seule classe PHP.

    Supposons une classe Calculator.php situé dans src/Support/. Notre classe à une seule fonction add() qui prends en paramêtre deux entiers et les additionne : 

    <?php

namespace App\Support;

class Calculator
{
    public function add(int $a, int $b): int
    {
        return $a + $b;
    }
}
?>

    On va créer une classe qui contiendra nos tests dans le répertoire tests/TestCase/Support/CalculatorTest.php. Par convention on va respecter l’arborescence des répertoires (/Support) et le nom de la classe test va être le même que celui de la classe testée à laquelle on ajoute le suffixe Test (Calculator deviendra CalculatorTest). La classe doit "étendre" (extends) TestCase qui, une classe de PHPUnit qui nous permettra de faire nos test. Voici un exemple de test où l'on s'assure que la fonction add() retourne le bon résultat.

    <?php
// CalculatorTest.php
namespace App\Test\TestCase\Support;

use App\Support\Calculator;
use PHPUnit\Framework\TestCase;

// Oubliez pas d'ajouter extends TestCase
class CalculatorTest extends TestCase
{
    public function testAddReturnValidSum(): void
    {
        // Création d'un objet Calculator
        $calculator = new Calculator();
        // Exécution de la fonction add
        $result = $calculator->add(30, 12);

        // Ici on "assert" que la fonction add de la 
        // classe Calculator retourne la bonne valeur
        $this->assertEquals(42, $result);
    }
}

    Et maintenant pour lançons notre test avec la commande composer test à partir d'une console à la racine de notre projet.

    Résultat positif

    Si on lance le test avec notre exemple, on va avoir le résultat suivant. On peut voir que la fonction testAddReturnValidSum retourne un résultat valide.

    C’est le nom de la fonction qui est utilisé comme titre dans le résultat des tests. Il faut qu’elle commence par test et qu’on utilise le « CamelCase » testAddReturnValidSum => Add return valid sum.

    Tests unitaires de nos routes

    On va créer une classe par route qu’on veut tester. Elle sera créer dans le répertoire tests/TestCase/[La même structure que la classe à tester]

    Exemple d'une classe test

    Je veux un test qui couvre la classe src/Action/Book/BookViewByIdAction.php, je vais créer ma classe test dans le répertoire tests/TestCase/Action/Book. On pourrait seulement la créer dans tests/TestCase mais la première structure permet de garder le code claire, surtout si on a beaucoup de tests à créer.

    • La classe test va porter le même nom que la classe testé en y ajoutant Test à la fin.
    • Je veux tester BookViewByIdAction, ma classe test va donc se nommer BookViewByIdActionTest.
    • La classe va « extends » TestCase et utiliser AppTestTrait (voir code exemple plus loin).
    • Ensuite on va crée une ou des fonctions qui vont contenir nos tests.
    • Le nom de la fonction doit obligatoirement commencer par test.
    • Le nom de la fonction sera ce qui est affiché dans le résultat. Le mot test du début sera supprimé et le caractère _ deviendra un espace.
    • testBookViewById_Retourne_les_bonnes_valeurs devient BookViewById Retourne les bonnes valeurs
    <?php

namespace App\Test\TestCase\Action\Book;

use App\Test\Traits\AppTestTrait;
use PHPUnit\Framework\TestCase;

class BookViewByIdActionTest extends TestCase
{
    // On inclus ici une référence au trait qu'on a créé plus haut
    use AppTestTrait;

    public function testBookViewById_Retourne_les_bonnes_valeurs(): void
    {
        // Les valeurs de retour attendues
        $expected = [
            [
                "id" => "74",
                "genreId" => "1",
                "titre" => "Blade Runner",
                "isbn" => "9782290314944"
            ]
        ];

        // On crée la requête en faisant un appel à la route à tester
        // la fonction createJsonRequest provient du trait AppTestTrait
        $request = $this->createJsonRequest('GET', '/books/74');
        // On effectue la requête et récupère le résultat
        $response = $this->app->handle($request);

        // C'est ici qu'on fait nos affirmations (assert)
        // J'affirme que le code de statut de la réponse est 200
        $this->assertSame(200, $response->getStatusCode());
        // J'affirme que les valeurs de retour correspondent à ce qui est attendu
        $this->assertJsonData($expected, $response);
    }
}
